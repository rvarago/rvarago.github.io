* Introdução

A *testabilidade* é um aspecto fundamental no desenvolvimento de sistemas embarcados, especialmente devido ao impacto significativo que esses dispositivos exercem em contextos críticos, como veículos, equipamentos médicos e sistemas industriais. Nesse sentido, testes unitários desempenham um papel estratégico ao validar isoladamente cada unidade de código, contribuindo para a detecção precoce de defeitos, reduzindo os riscos associados à integração de componentes e fornecendo feedback contínuo durante o desenvolvimento.

Entretanto, o ambiente embarcado apresenta desafios específicos, como restrições de hardware, recursos computacionais limitados e dificuldades em simular cenários críticos. Esses fatores destacam ainda mais a necessidade de uma abordagem estruturada para realizar testes eficazes.

Neste artigo, exploraremos estratégias práticas para isolar e testar subsistemas embarcados, com ênfase na aplicação de técnicas de injeção de dependências. Essas técnicas são especialmente úteis em sistemas legados, cuja arquitetura inicial pode não ter considerado a testabilidade uma prioridade.

Os exemplos apresentados ao longo do artigo serão ilustrados em C++.

* Motivação

Ao realizar um teste, o objetivo é validar, com certo grau de confiança, que o sistema atende aos requisitos definidos. A eficácia de um teste pode ser analisada com base em diversos critérios, notadamente:

+ *Fidelidade*: Refere-se ao quão próximo o ambiente de teste reflete as condições reais de operação.
+ *Reprodutibilidade*: Representa a consistência dos resultados obtidos em execuções sob as mesmas condições.
+ *Isolamento*: A habilidade de testar um subsistema de forma independente do restante do sistema.

Por vezes, esses critérios podem entrar em conflito. Por exemplo, um teste com alta fidelidade pode envolver variáveis externas que afetem negativamente tanto a reprodutibilidade quanto o isolamento. Imagine testar o envio de dados de um cliente IoT para um servidor real (alta fidelidade). Se o servidor estiver indisponível ou sofrer alterações, o resultado do teste pode se tornar incerto (baixa reprodutibilidade) ou exigir uma coordenação complexa com infraestrutura externa (baixo isolamento).

Essas considerações evidenciam a importância de uma estratégia robusta de testes, que abranja desde unidades isoladas até a integração completa de todos os subsistemas.

Neste artigo, o foco será a aplicação da *injeção de dependências* para viabilizar testes unitários em C++.

* Injeção de dependências

No contexto deste artigo, uma "dependência" representa um componente de software necessário para implementar um requisito. Exemplos incluem:

+ Uma função que realiza a leitura de um sensor de temperatura.
+ Um conjunto de classes que processa bytes de uma porta serial e os interpreta como logins RFID.
+ Um SDK para envio de telemetria.
+ Um controlador PID.

Nem sempre temos controle total sobre nossas dependências, seja porque elas são bibliotecas externas, seja porque envolvem acesso direto ao hardware. Nessas situações, testá-las em ambientes reais pode ser inviável ou inconveniente, especialmente em sistemas embarcados onde o hardware pode não estar disponível durante o desenvolvimento ou integração.

Assim como usamos modelos para representar a realidade, desejamos substituir uma dependência real por uma versão simplificada, permitindo que o componente seja testado de forma isolada. Em resumo, queremos isolar e controlar dependências que introduzam indeterminação ou efeitos colaterais fora do nosso controle, quando o objetivo é testar apenas o subsistema em questão.

É isso que a injeção de dependências possibilita.

A essência dessa abordagem é fazer com que o subsistema sob teste declare suas dependências como abstrações (em vez de ligações fixas, amarradas internamente ao subsistema), delegando a tarefa de fornecer implementações concretas ao componente externo que o utilizará. Dessa forma, o código sob teste recebe suas dependências de forma "injetada", permitindo que diferentes implementações sejam fornecidas em diferentes contextos. Essa configuração geralmente ocorre próximo ao ponto de entrada do software, como na função ~main~ ou equivalente.

Dessa forma, os testes podem definir seus próprios objetos substitutos que implementem a abstração de forma conveniente para verificar comportamentos específicos.

*** Considerações sobre Fakes, Mocks e Stubs

Há diferentes tipos de substitutos para testes, e a terminologia é [[https://martinfowler.com/articles/mocksArentStubs.html][diversa]] e, por vezes, confusa. Entre os termos comuns, destacam-se:
+ *Fake*: Uma implementação funcional simplificada, usada em testes.
+ *Stub*: Um substituto que retorna valores pré-definidos, sem conter lógica.
+ *Mock*: Permite verificar interações, como número de chamadas e argumentos usados.

Neste texto, utilizaremos o termo /Fake/ de forma genérica, sem rigor em relação ao seu significado estrito.

* O subsistema a ser testado

Para ilustrar os conceitos, considere uma API para envio de requisições HTTP:

#+BEGIN_SRC cpp
namespace http {
struct Request;

enum class Status {
  Okay,
  Fail
};

Status post(const Request&);
}
#+END_SRC

Essa API é imutável, pois pode fazer parte de um SDK proprietário ou de terceiros, logo não temos controle sobre seu comportamento nem podemos alterá-la.

Agora, considere o seguinte subsistema que utiliza essa API para enviar medições:

#+BEGIN_SRC cpp
struct Measurement {
  // Fictício.
};

http::Request to_request(const Measurement& m) {
  // Conversão fictícia para exemplo.
  return http::Request{/*...*/};
}

void send_measurement(const Measurement& measurement, int max_attempts) {
  const auto request = to_request(measurement);

  for (int attempt = 0; attempt < max_attempts; ++attempt) {
    const auto status = http::post(request);
    if (status == http::Status::Okay) {
      return;
    }
  }
}
#+END_SRC

Nosso objetivo é testar a função ~send_measurement~, garantindo que ela envie uma requisição exatamente ~N~ vezes em caso de ~N~ falhas consecutivas nas chamadas HTTP. Para tanto, objetivamos minimizar alterações no código original, de forma a garantir a retrocompatibilidade com código que já utiliza essa função.

Ao analisar a estratégia de testes unitários, percebemos que usar a API HTTP real nos testes traria os seguintes inconvenientes:

+ Dependência de um servidor externo, que pode estar indisponível.
+ Dificuldade em simular falhas específicas do servidor.
+ Testes lentos e dispendiosos devido à utilização de infraestrutura real.

Para resolver isso, propomos substituir a dependência direta da função ~http::post~ por uma versão simulada (/Fake/) projetada especificamente para testes.

* Técnicas de Injeção de Dependências

** Classes abstratas e resolução dinâmica

Uma abordagem inicial faz uso de classes abstratas para permitir polimorfismo dinâmico:

#+BEGIN_SRC cpp
class Client {
public:
  virtual ~Client() = default;
  virtual http::Status post(const http::Request&) const = 0;
};

class HttpClient : public Client {
public:
  http::Status post(const http::Request& req) const override {
    return http::post(req);
  }
};

void send_measurement(const Measurement& measurement, int max_attempts,
                      const Client& client = HttpClient{}) {
  const auto request = to_request(measurement);

  for (int attempt = 0; attempt < max_attempts; ++attempt) {
    const auto status = client.post(request);
    if (status == http::Status::Okay) {
      return;
    }
  }
}

// Fake para testes: conta tentativas e retorna status fixo.
class CountingClient : public Client {
public:
  explicit CountingClient(http::Status fixedStatus) : status{fixedStatus} {}

  http::Status post(const http::Request&) const override {
    ++attempts;
    return status;
  }

  mutable int attempts{0}; // "mutable" permite modificar o estado dentro de funções-membra "const". Perigoso em código de produção, porém útil em testes.
  http::Status status;
};

void test() {
  constexpr int expected_attempts = 10;
  CountingClient client_always_fail{http::Status::Fail};

  send_measurement(Measurement{}, expected_attempts, client_always_fail);

  assert(client_always_fail.attempts == expected_attempts);
}
#+END_SRC

Note que ~send_measurement~ agora recebe um terceiro parâmetro: ~const Client& client = HttpClient{}~. É ele que nos possibilita substituir a real chamada HTTP por algo que esteja sob nosso controle, permitindo testes determinísticos e isolados. Sendo essa a essência da injeção de dependências.

Além disso, lhe demos um argumento padrão (por simplicidade, instanciado a cada chamada) para preservar a compatibilidade com o código existente. Apesar de haver outras formas com diferentes vantagens e desvantagens para se atingir o mesmo objetivo (por exemplo, sobrecarga de funções, passagem explícita de argumentos), o argumento padrão será utilizado como estratégia ao longo desse texto por simplicidade, sem perda de generalidade.

Já no teste, utilizamos um /Fake/ que simplesmente retorna uma resposta de falha pré-definida e mantém um contador do número de chamadas feitas ao método. Esse contador pode ser inspecionado para confirmar que corresponde ao valor esperado.

Note que /Fakes/ podem ser muito mais elaborados. Por exemplo, poderíamos substituir o contador por um contêiner (por exemplo, ~std::vector<http::Request>~) que armazena os argumentos fornecidos ao ~post~ e assim poderíamos verificar cenários em que ~send_measurement~ é chamada com diferentes argumentos. Neste caso, o número de chamadas equivaleria ao número de elementos no contêiner.

#+begin_center
O uso de ~mutable~ exige cautela, pois pode introduzir, de forma não intencional, violações de "[[https://isocpp.org/wiki/faq/const-correctness][const correctness]]". Ainda assim, seu emprego é um padrão reconhecido e amplamente utilizado em fixtures de teste, sendo considerado pragmático nesse contexto.
#+end_center

Considerações:
+ *Vantagens*:
  + Permite um uso mais eficiente da compilação incremental.
  + Baixo impacto no tamanho do binário.
+ *Desvantagens*:
  + Overhead de resolução dinâmica e possível perda de otimizações que dependem de /inlining/ (embora nem sempre, e.g. [[https://quuxplusone.github.io/blog/2021/02/15/devirtualization/][de-virtualização de chamadas]]).
  + Maior dependência do uso de ponteiros e referências.

** Caso especial: ~std::function<>~

Para casos em que a API resume-se a uma única função, pode-se usar [[https://en.cppreference.com/w/cpp/utility/functional/function.html][~std::function<>~]] para definir o comportamento dinamicamente sem necessidade de hierarquia de classes, o que torna a implementação menos verbosa:

#+BEGIN_SRC cpp
// Camada de indireção para substituição dinâmica.
using PostFn = std::function<http::Status(const http::Request&)>;

void send_measurement(const Measurement& measurement, int max_attempts,
                      const PostFn& post = http::post) {
  const auto request = to_request(measurement);

  for (int attempt = 0; attempt < max_attempts; ++attempt) {
    const auto status = post(request);
    if (status == http::Status::Okay) {
      return;
    }
  }
}

class CountingClient {
public:
  explicit CountingClient(http::Status fixedStatus) : status{fixedStatus} {}

  // Implementamos o operador chamada de função para que "CountingClient" seja compatível com "std::function".
  http::Status operator()(const http::Request &) const {
    ++attempts;
    return status;
  }

  http::Status status;
  mutable int attempts{0};
};

void test() {
  constexpr int expected_attempts = 10;
  CountingClient client_always_fail{http::Status::Fail};

  send_measurement(Measurement{}, expected_attempts,
                   std::cref(client_always_fail));

  assert(client_always_fail.attempts == expected_attempts);
}
#+END_SRC

Note o uso de [[https://en.cppreference.com/w/cpp/utility/functional/ref.html][~std::cref~]] para impedir a cópia de ~client_always_fail~. Isso garante que seja passado por referência, permitindo que qualquer modificação feita em ~client_always_fail~ dentro da função ~send_measurement~ seja refletida após sua execução, facilitando a inspeção dos resultados.

Alternativamente, poderíamos implementar o /Fake/ como uma expressão lambda, sem a necessidade de uma classe, por exemplo:

#+BEGIN_SRC cpp
int attempts = 0;
const auto client_always_fail = [&attempts](const http::Request &)  {
  ++attempts;
  return http::Status::Fail;
};

send_measurement(Measurement{}, expected_attempts, client_always_fail);

assert(attempts == expected_attempts);
#+END_SRC

Considerações:
+ *Vantagens*:
  + Mais simples que classes abstratas.
  + Reduz verbosidade.
+ *Desvantagens*:
  + Overhead de ~std::function~ pode ser relevante em ambientes embarcados restritos e/ou sem acesso ao "heap".
  + Requer refatoração para acomodar múltiplas funções.

** Templates e resolução estática

Uma alternativa com *templates* explora polimorfismo estático, resolvendo chamadas em tempo de compilação:

#+BEGIN_SRC cpp
class HttpClient {
public:
  http::status post(const http::Request &req) const { return http::post(req); }
};

// Nossa camada de indireção passa a ser o template.
template <typename Client = HttpClient>
void send_measurement(const Measurement &measurement, int max_attempts,
                      const Client &client = HttpClient{}) {
  const auto request = to_request(measurement);

  for (int attempt = 0; attempt < max_attempts; ++attempt) {
    const auto status = client.post(request);
    if (status == http::Status::Okay) {
      return;
    }
  }
}

class CountingClient {
public:
  explicit CountingClient(http::Status fixedStatus) : status{fixedStatus} {}

  http::Status post(const http::Request &) const {
    ++attempts;
    return status;
  }

  http::Status status;
  mutable int attempts{0};
};

void test() {
  constexpr int expected_attempts = 10;

  CountingClient client_always_fail{http::Status::Fail};
  send_measurement(Measurement{}, expected_attempts, client_always_fail);

  assert(client_always_fail.attempts == expected_attempts);
}
#+END_SRC

Considerações:
+ *Vantagens*:
  + Maior eficiência em tempo de execução.
  + Mais favorável ao /inlining/ e às otimizações dele decorrentes.
+ *Desvantagens*:
  + Aumento no tamanho do binário (o que tambem pode after o cache de instruções e assim o desempenho).
  + Aumento no tempo de compilação.
  + Complica a compilação em unidades de compilação separadas, pois para compilar cada instanciação do template o compilador requer acesso à sua definição completa.
  + Qualidade das mensagens de erro de compilação pode ser prejudicada.

** Caso especial: função como template

#+BEGIN_SRC cpp
// Nossa camada de indireção passa a ser algo "tipo" uma função.
template <typename PostFn = decltype(&http::post)>
void send_measurement(const Measurement &measurement, int max_attempts,
                      const PostFn& post = http::post) {
  const auto request = to_request(measurement);

  for (int attempt = 0; attempt < max_attempts; ++attempt) {
    const auto status = post(request);
    if (status == http::Status::Okay) {
      return;
    }
  }
}

class CountingClient {
public:
  explicit CountingClient(http::Status fixedStatus) : status{fixedStatus} {}

  // Implementamos o operador chamada de função para que "CountingClient" seja compatível com o template.
  http::Status operator()(const http::Request &) const {
    ++attempts;
    return status;
  }

  http::Status status;
  mutable int attempts{0};
};

void test() {
  constexpr int expected_attempts = 10;
  CountingClient client_always_fail{http::Status::Fail};

  send_measurement(Measurement{}, expected_attempts,
                   std::cref(client_always_fail));

  assert(client_always_fail.attempts == expected_attempts);
}
#+END_SRC

Similar ao ~std::function<>~, podemos também alcançar o mesmo objetivo com uma expressão lambda em lugar da classe ~CountingClient~.

Considerações:
+ *Vantagens*:
  + Mais simples que classes, reduz verbosidade.
+ *Desvantagens*:
  + Similares ao uso de templates.
  + Requer refatoração para múltiplas funções.

*** Conceitos (C++ >=20)

Citamos a diminuição da qualidade dos erros de compilação como sendo uma desvantagem dos templates.

Isso é parcialmente mitigado com o uso de "[[https://en.cppreference.com/w/cpp/language/constraints.html][concepts]]" (recurso disponível desde o C++20), que permitem definir "interfaces estáticas" de maneira explícita. Assim, podemos, por exemplo, restringir nosso template a tipos que: suportem o operador de chamada de função, aceitem um ~http::Request~ como parâmetro e que retornem um ~http::Status~, como no exemplo a seguir:

#+begin_src cpp
template <typename F>
concept PostFunction =
    std::invocable<F, const http::Request&> &&
    std::same_as<std::invoke_result_t<F, const http::Request&>, http::Status>;

* Conclusão

A injeção de dependências é uma técnica essencial para melhorar a testabilidade de software embarcado, permitindo isolar e simular componentes de maneira eficiente. As abordagens disponíveis oferecem diferentes vantagens e custos:

- *Classes abstratas*: Proporcionam flexibilidade em tempo de execução, mas introduzem overhead de resolução dinâmica.
- *Templates*: Garantem alto desempenho e resoluções estáticas, mas podem ter impacto negativo no tamanho do binário e no tempo de compilação.

O impacto de cada abordagem no projeto, como viabilidade e desempenho, depende de diversos fatores que devem ser analisados e monitorados constantemente. É fundamental evitar comparações simplistas entre técnicas, uma vez que cada solução apresenta características únicas e é mais adequada para contextos específicos. Por isso, recomenda-se uma avaliação criteriosa caso a caso, considerando ainda a possibilidade de combinar técnicas para atender às necessidades reais do projeto.

Concluindo, no que tange à estratégia de testes, a escolha da abordagem mais adequada deve considerar as particularidades do sistema, buscando equilibrar fidelidade aos comportamentos reais, isolamento das dependências e reprodutibilidade dos resultados. Vale destacar que, ao utilizar /fakes/, o nível de confiança nos testes estará diretamente relacionado à proximidade do /fake/ com o comportamento e as características da dependência real. /Fakes/ excessivamente simplificados podem introduzir discrepâncias significativas, comprometendo a validade dos testes ao longo de sua vida útil. Portanto, é essencial investir na (i) criação de /fakes/ suficientemente realistas, que aumentem a confiabilidade dos testes, e (ii) execução de testes que integrem o subsistema com suas dependências reais para validar os cenários de maneira mais abrangente.

Particularmente em sistemas embarcados, a escolha de estratégias que minimizem os impactos do hardware e reforcem a confiabilidade do software é essencial. Essa abordagem pode incluir desde o uso de testes automatizados que exploram uma ampla gama de cenários simulados, até testes manuais minuciosos que validem o comportamento do sistema em condições reais. Um processo contínuo de revisão e refinamento das estratégias de teste, aliado à aplicação adequada da injeção de dependências, garante uma base sólida para a evolução e a confiabilidade do software embarcado, contribuindo para sistemas mais seguros e confiáveis.
